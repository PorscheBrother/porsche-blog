Kubernetes 完整教學

由 Porsche哥 編寫 • 2026年4月2日

什麼是 Kubernetes？

Kubernetes（通常縮寫為 K8s）是一個開源的容器編排平台，由 Google 原始開發，現在由 Cloud Native Computing Foundation (CNCF) 維護。它能夠自動化部署、擴容和管理容器化應用程式。

K8s 名稱來源：Kubernetes 有 8 個字母在 "K" 和 "s" 之間，因此 K8s 代表 "K" + 8 個字母 + "s"

為什麼需要 Kubernetes？

在微服務架構和容器化時代，傳統的部署方式無法滿足快速迭代和彈性擴容的需求。Kubernetes 提供了：

  • 自動部署與回滾： Declarative 配置確保應用程式始終處於預期狀態
  • 服務發現與負載均衡： 自動分配 IP 和 DNS 名稱，負載流量分配
  • 儲存空間編排： 自動掛載本地儲存、公有雲提供者等
  • 自動擴縮容： 根據 CPU 使用率或其他自定義指標調整 Pod 數量
  • 自我修復： 重新啟動失敗的容器，替換和重新排程節點
  • 密鑰與配置管理： 安全地存儲和管理敏感資訊如密碼、OAuth 權杖和 SSH 金鑰

Kubernetes 基礎概念

節點 (Nodes)

節點是運行容器的工作機器。可以是實體機器或虛擬機器。每個節點包含運行 Pods 所需的服務：

  • Container Runtime： 負責運行容器（如 Docker、containerd、cri-o）
  • Kubelet： 在每個節點上的代理，確保容器在 Pod 中運行
  • Kube-proxy： 維護網路規則並執行連線轉發

Pods

Pod 是 Kubernetes 中最小的部署單位。它代表叢集中正在運行的一個或多個容器。Pod 中的容器共享:

  • 網路命名空間（同一 IP 和埠空間）
  • 掛載卷（共享檔案系統）
  • 生命週期（一起被創建和銷毀）
# 最簡單的 Pod YAML apiVersion: v1 kind: Pod metadata: name: hello-world-pod labels: app: hello-world spec: containers: - name: hello-world image: nginx:latest ports: - containerPort: 80

命名空間 (Namespaces)

命名空間提供了一種將叢集資源隔離的機制。在同一個叢集中，您可以創建多個命名空間來邏輯上分隔不同的團隊、專案或環境。

# 創建命名空間 apiVersion: v1 kind: Namespace metadata: name: development

標籤 (Labels) 和選擇器 (Selectors)

標籤是附加到物件上的鍵值對，用於組織和選擇子集物件。選擇器則用於根據標籤過濾物件。

# 帶標籤的 Pod apiVersion: v1 kind: Pod metadata: name: labeled-pod labels: app: web tier: frontend release: stable spec: containers: - name: web image: nginx

控制器 (Controllers)

控制器是確保叢集處於預期狀態的控制循環。它持續監視當前狀態並採取行動使其達到預期狀態。

ReplicaSet

ReplicaSet 確保在任何時候都有指定數量的 Pod 副本在運行。如果 Pod 數量不足，它會創建新的 Pod；如果過多，它會終止多餘的 Pod。

# ReplicaSet 範例 apiVersion: apps/v1 kind: ReplicaSet metadata: name: frontend-replicaset labels: app: guestbook tier: frontend spec: replicas: 3 selector: matchLabels: tier: frontend template: metadata: labels: tier: frontend spec: containers: - name: php-redis image: gcr.io/google_samples/gb-frontend:v3

Deployment（推薦使用）

Deployment 提供聲明式更新 Pod 和 ReplicaSet 的能力。您只需要描述預期狀態，Deployment 控制器會逐步將實際狀態變更為預期狀態。

# Deployment 範例 apiVersion: apps/v1 kind: Deployment metadata: name: nginx-deployment labels: app: nginx spec: replicas: 3 selector: matchLabels: app: nginx template: metadata: labels: app: nginx spec: containers: - name: nginx image: nginx:1.14.2 ports: - containerPort: 80

服務 (Service)

Service 是一種抽象概念，它定義了一種邏輯上的 Pod 集合和訪問這些 Pod 的策略。Service 使應用程式可以透過單一的入口點訪問後端 Pod，即使 Pod 會因為擴縮容或失敗而變化。

Service 類型

  • ClusterIP： 預設類型，僅在叢集內部可訪問
  • NodePort： 在每個節點的靜態埠上暴露服務
  • LoadBalancer： 使用雲提供者的負載均衡器暴露服務
  • ExternalName： 返回 CNAME 和其值，將服務映射到 externalName 資料內容
# Service 範例（ClusterIP） apiVersion: v1 kind: Service metadata: name: nginx-service spec: selector: app: nginx ports: - protocol: TCP port: 80 targetPort: 80 type: ClusterIP

配置與密鑰管理

ConfigMap

ConfigMap 用於存儲非機密性的配置數據。Pod 可以將 ConfigMap 作為環境變數、命令行參數或掛載卷的方式使用。

# ConfigMap 範例 apiVersion: v1 kind: ConfigMap metadata: name: game-demo data: # 屬性鍵值對 player_initial_lives: "3" ui_properties_file_name: "ui.properties" # 檔案類似資料 game.properties: | enemy.types=aliens,monsters player.maximum-lives=5 ui.properties: | color.good=purple color.bad=yellow allow.textmode=true

Secret

Secret 用於存儲和管理敏感資訊，如密碼、OAuth 權杖和 SSH 金鑰。將這樣的資訊放在 Secret 中比放在 Pod 定義或 Docker Image 中更安全和更靈活。

# Secret 範例（建議使用 kubectl create secret 建立） apiVersion: v1 kind: Secret metadata: name: db-secret type: Opaque data: # 值必須是 base64 編碼 username: YWRtaW4= password: MWYyZDFlMmU2N2Rm

存儲卷 (Volumes)

Volume 解決了容器檔案系統的兩個問題：

  • 當容器崩潰時， kubelet 會重新啟動它，但容器中的檔案會丟失
  • 當需要在容器之間共享檔案時

Kubernetes 支持多種卷類型，包括本地磁碟、雲端硬碟、網路檔案系統等。

# 使用 emptyDir 卷的 Pod apiVersion: v1 kind: Pod metadata: name: test-empty-dir spec: containers: - name: test-container image: nginx volumeMounts: - name: cache-volume mountPath: /cache volumes: - name: cache-volume emptyDir: {}

命名空間資源配額

為防止資源耗盡，您可以為命名空間設置資源配額，限制該命名空間內可以消耗的資源數量。

# 資源配額範例 apiVersion: v1 kind: ResourceQuota metadata: name: compute-quota namespace: development spec: hard: pods: "10" requests.cpu: "4" requests.memory: 5Gi limits.cpu: "10" limits.memory: 10Gi

實踐：部署您的第一個應用

讓我們一步步部署一個簡單的網站應用。

步驟 1：建立 Deployment

# 建立 hello-deployment.yaml apiVersion: apps/v1 kind: Deployment metadata: name: hello-deployment spec: replicas: 2 selector: matchLabels: app: hello template: metadata: labels: app: hello spec: containers: - name: hello image: nginx:latest ports: - containerPort: 80 resources: requests: memory: "64Mi" cpu: "250m" limits: memory: "128Mi" cpu: "500m"

步驟 2：暴露為 Service

# 建立 hello-service.yaml apiVersion: v1 kind: Service metadata: name: hello-service spec: selector: app: hello ports: - protocol: TCP port: 80 targetPort: 80 type: LoadBalancer

步驟 3：應用配置

# 使用 kubectl 應用配置 kubectl apply -f hello-deployment.yaml kubectl apply -f hello-service.yaml # 檢查部署狀態 kubectl get deployments kubectl get pods kubectl get services

進階概念

Helm 套件管理器

Helm 是 Kubernetes 的套件管理器，類似於 apt/yum/Homebrew。它使用圖表（Charts）來定義、安裝和升級複雜的 Kubernetes 應用。

使用 Helm 的好處：

  • 可重複部署應用
  • 版本控制和回滾
  • 相依性管理
  • 模板化配置

監控與日誌

生產環境中監控和日誌記錄至關重要。Kubernetes 生態系統提供了豐富的工具：

  • 監控： Prometheus + Grafana
  • 日誌： ELK 棧 (Elasticsearch, Logstash, Kibana) 或 Fluentd
  • 追蹤： Jaeger 或 Zipkin

最佳實踐

  • 使用命名空間： 隔離不同環境和團隊
  • 資源請求與限制： 為所有容器設置適當的 requests 和 limits
  • 健康檢查： 使用 liveness 和 readiness 探針
  • 滾動更新： 使用 Deployment 進行無縫更新
  • 備份與災難復原： 定期備份 etcd 和持久卷
  • 安全性： 使用 RBAC、Pod 安全標準和網路策略

結語

Kubernetes 看起來可能很複雜，但一旦理解了其核心概念和設計哲學，您會發現它是一個極其強大且靈活的平台。從基本的 Pod 到複雜的微服務架構，Kubernetes 提供了構建現代雲原生應用所需的所有工具。

隨著實踐經驗的增加，您將能夠熟練地使用 Kubernetes 來構建、部署和管理可擴展、彈性和自愈的應用程式。

記住：Kubernetes 是一個生態系統，不是孤立的工具。持續學習和探索相關工具（如 Istio、Knative、Argo CD 等）將使您的 Kubernetes 之旅更加豐富。
